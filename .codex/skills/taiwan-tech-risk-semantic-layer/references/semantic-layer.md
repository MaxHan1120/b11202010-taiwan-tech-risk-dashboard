# Taiwan Tech Risk Semantic Layer

## Quick Reference

- Area: Taiwan Semiconductor & Tech Stock Risk Dashboard
- Intended users: Retail-investor dashboard users, course graders, and future report-writing agents
- Coverage level: Directional
- Source inventory: `references/source-inventory.md`
- Last synthesized: 2026-06-19
- Freshness expectations: market data is designed for daily trading-day refresh after market close; monthly revenue and financial statements are checked daily but update monthly or quarterly.
- Default date and time zone rules: use Taiwan market dates for stock observations; API `updated_at` values are UTC timestamps generated by the backend.

## Key Metrics

| Metric | Definition | Numerator | Denominator | Time Grain | Canonical Source | Caveats |
| --- | --- | --- | --- | --- | --- | --- |
| `total_score` | Weighted rule-based score: 25% volatility, 25% drawdown, 20% abnormal volume, 15% valuation, 15% fundamental | Weighted component scores | 100-point scale | Latest available trading date per stock | `backend/app/processing/risk.py` | Monitoring signal only; not investment advice |
| `volatility_score` | Percentile rank of latest 20-day annualized volatility versus the stock's available history | Count of historical rolling vol values below or equal to latest | Number of valid rolling vol observations | Daily | `backend/app/processing/risk.py` | Defaults to 35 when insufficient data |
| `drawdown_score` | Current drawdown from the 60-day high, scaled so 35% drawdown maps to 100 | 60-day high minus latest close | 60-day high | Daily | `backend/app/processing/risk.py` | Capped to 0-100 |
| `volume_score` | Latest volume spike relative to 20-day average volume | Latest volume divided by 20-day average volume | Scale where 3x average maps to 100 | Daily | `backend/app/processing/risk.py` | Defaults to 35 with fewer than 20 observations |
| `valuation_score` | Average percentile rank of latest PER and PBR versus available valuation history | PER and PBR percentile scores | Two valuation components when available | Daily or available valuation dates | `backend/app/processing/risk.py` | Defaults to 35 when valuation data is missing |
| `fundamental_score` | Average risk from revenue YoY weakness and recent EPS/gross profit/net income declines | Component deterioration signals | Available fundamental components | Monthly or quarterly source cadence | `backend/app/processing/risk.py` | Uses latest available monthly or quarterly data; not daily fundamental changes |
| `risk_level` | Bucket derived from `total_score` | `total_score` | Thresholds: Low 0-39, Medium 40-69, High 70-100 | Daily score output | `backend/app/processing/risk.py`, `README.md` | Thresholds are prototype defaults |

## Query Patterns

- Pattern: Latest risk ranking
  - Use when: answering which tracked stocks are highest risk now.
  - Key tables: `risk_scores`, `stocks`, optionally `daily_prices`.
  - Required filters: latest `risk_scores.date` per `stock_id`.
  - Common joins: `stocks.stock_id = risk_scores.stock_id`; `daily_prices` on same `stock_id` and `date`.
  - Example skeleton: select latest risk rows per stock and order by `total_score desc`.

- Pattern: Stock detail page
  - Use when: explaining one stock's price trend, current risk score, valuation, revenue, and financials.
  - Key tables: `stocks`, `daily_prices`, `risk_scores`, `valuation_metrics`, `monthly_revenue`, `financial_metrics`.
  - Required filters: one `stock_id`; latest rows for summary and ordered history for charts.
  - Common joins: mostly independent lookups by `stock_id`; join by date only when price and risk share a scoring date.

- Pattern: Portfolio analysis
  - Use when: interpreting a user-provided watchlist or holdings weights.
  - Key tables: `stocks`, latest `risk_scores`.
  - Required filters: user-provided stock IDs in the fixed universe.
  - Common joins: latest `risk_scores` per stock joined to `stocks`.

## Gotchas

- Gotcha: Risk score is not a forecast.
  - Impact: Future answers must not claim the system predicts returns or recommends trades.
  - How to avoid: describe it as a monitoring, screening, and research-efficiency tool.
  - Source: `README.md`, product positioning in user plan.

- Gotcha: Demo data is deterministic fallback.
  - Impact: Demo values can support UI and pipeline testing but should not be cited as live market facts.
  - How to avoid: check whether data came from `seed_demo_data.py`, `update_log`, or FinMind update path before making current-market claims.
  - Source: `backend/app/ingestion/seed.py`, `backend/app/main.py`.

- Gotcha: Fundamental metrics update less frequently than market data.
  - Impact: Daily risk scores may combine current market data with latest monthly or quarterly fundamentals.
  - How to avoid: state source cadence whenever explaining daily score changes.
  - Source: `README.md`, `backend/app/processing/risk.py`.
